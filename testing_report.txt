Testing, Verification & dmesg Output Analysis
=============================================


3.1 Kernel Module Lifecycle
---------------------------
Purpose: Verify the module compiles, loads, creates the device node, and unloads cleanly.

3.1.1 Compilation

Run and capture terminal output:

    make clean && make

Include: Screenshot of full make output showing CC [M] and LD [M] lines with zero warnings.
Confirm led_driver.ko is produced.

3.1.2 Module Loading

    sudo insmod led_driver.ko
    echo "Exit code: $?"
    lsmod | grep led_driver
    ls -la /dev/gpioled
    sudo chmod 666 /dev/gpioled
    ls -la /dev/gpioled

Include: Screenshot showing lsmod output with led_driver listed, and ls -la showing
/dev/gpioled before and after chmod 666.

3.1.3 Module Unloading

    sudo rmmod led_driver
    echo "Exit code: $?"
    ls /dev/gpioled 2>&1
    lsmod | grep led_driver || echo "Module not listed — successfully removed"

Include: Screenshot confirming /dev/gpioled no longer exists after rmmod.

3.1.4 Error Cases

    # Double-insert (while already loaded)
    sudo insmod led_driver.ko

    # Remove when not loaded
    sudo rmmod led_driver
    sudo rmmod led_driver

Include: Screenshot of both error messages.
Expected: "insmod: ERROR: could not insert module led_driver.ko: File exists"
          "rmmod: ERROR: Module led_driver is not currently loaded"

3.3 dmesg Output Analysis — Device File Operations
---------------------------------------------------
Purpose: Show that every file operation on /dev/gpioled produces the correct printk message,
confirming kernel-side handlers are called.

Run each command individually and capture dmesg | tail -5 after each.

3.3.1 Write "ON"

    echo "ON" | sudo tee /dev/gpioled
    dmesg | tail -5

Expected dmesg output:
    gpioled: device opened by user-space process
    gpioled: command 'ON'  received — GPIO 24 HIGH
    gpioled: device closed by user-space process

Include: Screenshot. Annotate: tee triggers open() -> dev_open(), write() -> dev_write()
(which calls gpio24_set(1), writing 1 << 24 to GPSET0), then close() -> dev_release().
Physical LED should be on.

3.3.2 Read State

    cat /dev/gpioled
    dmesg | tail -5

Expected terminal output: LED:ON
Expected dmesg output:
    gpioled: device opened by user-space process
    gpioled: state sent to user space -> LED:ON
    gpioled: device closed by user-space process

Include: Screenshot. Annotate: cat invokes read() -> dev_read(), which calls copy_to_user()
to transfer "LED:ON\n" across the kernel/user boundary.

3.3.3 Write "OFF"

    echo "OFF" | sudo tee /dev/gpioled
    dmesg | tail -5

Expected dmesg output:
    gpioled: device opened by user-space process
    gpioled: command 'OFF' received — GPIO 24 LOW
    gpioled: device closed by user-space process

Include: Screenshot. Physical LED should be off.

3.3.4 Read State After OFF

    cat /dev/gpioled

Expected terminal output: LED:OFF
Include: Screenshot.

3.3.5 Invalid Input

    echo "BLAH" | sudo tee /dev/gpioled
    dmesg | tail -3

Expected dmesg output:
    gpioled: unknown command 'BLAH' (expected 'ON' or 'OFF')

Include: Screenshot. Note severity: KERN_WARNING. dev_write() returns -EINVAL;
tee reports a write error.


3.4 Character Device — System Call Round-Trip Verification
----------------------------------------------------------
Purpose: Prove write() -> kernel -> hardware -> read() -> user-space data path is correct.

    # Write ON, read back state — must match
    echo "ON"  | sudo tee /dev/gpioled && cat /dev/gpioled

    # Write OFF, read back state — must match
    echo "OFF" | sudo tee /dev/gpioled && cat /dev/gpioled

Include: Screenshot showing both pairs. Both reads must return the state consistent with
the prior write. Annotate: copy_from_user() in dev_write() and copy_to_user() in dev_read()
are the kernel functions responsible for data transfer across the privilege boundary.


3.5 BCM2711 GPIO Register Verification
---------------------------------------
Purpose: Document the hardware register layer that the kernel module controls directly.

Include a table:

  Register          | Physical address | Offset from base | Role in this driver
  BCM2711_GPIO_BASE | 0xFE200000       | —                | Physical base of GPIO peripheral
  GPFSEL2           | 0xFE200008       | +0x08            | Bits 12-14 set to 001 — configure GPIO 24 as output
  GPSET0            | 0xFE20001C       | +0x1C            | Write 1 << 24 to drive GPIO 24 HIGH (LED on)
  GPCLR0            | 0xFE200028       | +0x28            | Write 1 << 24 to drive GPIO 24 LOW (LED off)

Include: Annotated extract of gpio24_set() from led_driver.c lines 123-132 showing the
writel() calls. Include a photo or multimeter reading showing GPIO 24 at ~3.3V when ON
and ~0V when OFF (or note simulation mode if ioremap failed).
